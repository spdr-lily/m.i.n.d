# M.I.N.D Development Instructions

## Before Starting Any Work

### 1. Understand the Project Context
- **Objective**: Build a probabilistic diagnostic inference engine for mental health
- **Scope**: MVP phase focusing on mood disorders (Depression, Bipolar Disorder)
- **Compliance**: LGPD/GDPR compliant; human-in-the-loop only
- **Data**: UUID-based, PII completely separated from analytical layer

### 2. Database & Models
- **Primary Keys**: Always use UUID (LGPD compliance)
- **Timestamps**: Use `created_at`, `updated_at` audit columns
- **Soft Deletes**: Consider for clinical records (retention policies)
- **Migrations**: Use Alembic; version all schema changes

Example:
```python
from sqlalchemy import Column, String, Float, DateTime, Boolean
```

### 3. Services & Business Logic
- **Separation of Concerns**: Services handle business logic, repositories handle data access
- **Inference Logic**: Place in `app/ml/` (Bayesian calculations, probability weighting)
- **Orchestration**: Services coordinate repositories + ML pipelines

Example:

### 4. API Routes
- **Authentication**: Use `app/security/` for auth, token validation
- **Input Validation**: Pydantic schemas with Field constraints
- **Error Responses**: Include clinical context, not technical details
- **Logging**: Log all diagnoses for clinical audit trail

Example:
```python
# app/api/diagnoses.py
from fastapi import APIRouter, Depends, HTTPException
from app.schemas.diagnosis import ConsultationSchema, DiagnosisResponseSchema
from app.services.diagnosis_service import DiagnosisService
from app.security.auth import get_current_clinician

router = APIRouter(prefix="/diagnoses", tags=["diagnoses"])

@router.post("/calculate", response_model=DiagnosisResponseSchema)
async def calculate_diagnosis(
    consultation: ConsultationSchema,
    clinician = Depends(get_current_clinician),
    service: DiagnosisService = Depends()
):
    """
    Calculate probabilistic diagnosis for patient consultation.
    
    Clinical Disclaimer:
    - Results are probabilistic support only
    - Human clinician maintains diagnostic authority
    - For clinical use in supervised settings only
    """
    try:
        result = service.calculate_diagnosis(consultation)
        # Log for audit trail
        return result
    except DiagnosticCriteriaError as e:
        raise HTTPException(status_code=400, detail=str(e))
```

## LGPD/GDPR Compliance Checklist

- [ ] All patient identifiers use UUID (no names, DOB in PII table only)
- [ ] PII table separate from clinical/analytical tables
- [ ] Encryption used for sensitive fields (consultation notes)
- [ ] Retention policies documented in code comments
- [ ] Data deletion/anonymization procedures implemented
- [ ] Clinical audit trail for all diagnoses
- [ ] Access control enforced (clinician roles)
- [ ] No data exported without de-identification

```python
# LGPD Compliance: Patient name and contact info stored separately in ENCRYPTED PII table

```

## Common Patterns

### Pattern 1: Adding a New Disorder
1. Add disorder record to database (with DSM/ICD codes)
2. Create Pydantic schema in `app/schemas/`
3. Implement criteria evaluation in `app/ml/criteria_evaluator.py`
4. Add service method in `app/services/diagnosis_service.py`
5. Create API route in `app/api/diagnoses.py`
6. Write pytest fixtures and unit tests
7. Document DSM/ICD reference with citations

### Pattern 2: Implementing Psychometric Scale
1. Define scale metadata (items, scoring logic) in `app/models/assessment_scales.py`
2. Create Pydantic schema for responses in `app/schemas/assessments.py`
3. Implement scoring service in `app/services/assessment_service.py`
4. Add API route in `app/api/assessment_scales.py`
5. Validate scoring against published benchmarks
6. Test with known clinical data

### Pattern 3: Adding a Comorbidity Rule
1. Define rule logic in `app/ml/diagnosis_relationships.py`
2. Test with integration tests (ensure both disorders evaluated)
3. Document clinical rationale (e.g., "MDD cannot occur during manic episode")
4. Log application of rule for audit trail

## Red Flags — Ask Before Proceeding

❌ **Don't**: Present probabilistic diagnosis as definitive  
→ Always include "human review required" flags, confidence intervals

❌ **Don't**: Store patient names in clinical tables  
→ Always use UUID; keep PII completely separate

❌ **Don't**: Skip type hints or docstrings  
→ Clinical code requires clear documentation for audit/compliance

❌ **Don't**: Test with real patient data  
→ Use synthetic MIMIC-IV-like fixtures; never real medical records

❌ **Don't**: Forget audit trails on diagnoses  
→ Log all calculations with timestamps, clinician ID, rationale
