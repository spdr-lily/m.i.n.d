# M.I.N.D Project Agent Configuration

## Project Context
You are assisting with **M.I.N.D (Mental Intelligence & Network Data)** — a Clinical Decision Support System for mental health diagnosis. This is a hybrid Data Science + Psychiatry project focused on probabilistic diagnostic inference using DSM-5-TR and ICD-11 criteria, with 19 disorders, 20 psychometric scales (10 clinical + 10 neuropsychological), ML pipeline (12 models), and full LGPD compliance.

**Current Phase:** CDSS completo — ~192 transtornos (19 core), 21 escalas, 12 modelos ML (MLflow), DW star schema, segurança SSD, 3 camadas de validação clínica.

## Code Patterns & Conventions

### 1. Service-Repository Pattern
- **Repositories** (`app/repositories/`): Pure data access, no business logic
  - Example: `patient_repository.py` with `get_by_uuid()`, `create_consultation()`
- **Services** (`app/services/`): 24 serviços de negócio
  - Exemplo: `inference_service.py`, `integrity_service.py`, `assessment_service.py`

### 2. Pydantic Schemas (`app/schemas/`)
All API requests/responses use Pydantic v2 com validadores:
- `consultation.py`: validadores para consultation_date, intensity 0-10, frequency enum, duration_days ≥ 1
- `patient_profile.py`: validador birth_date future reject
- `professional.py`: assigned_patient_uuids para atribuição de pacientes
- `disorder.py`: ClassificationAuthorityResponse, DSM-column fields

### 3. SQLAlchemy Models (`app/models/base.py`)
Todas as entidades em `base.py` com UUID PKs (LGPD):
- `ClassificationAuthority` — WHO, APA
- `ProfessionalPatientAssignment` — atribuição profissional-paciente
- `Disorder` com colunas: dsm_criteria, dsm_exclusions, dsm_differentials, icd11_exclusions, icd11_differentials
- `ICD11Code` com authority_id FK para WHO
- DB CHECK constraints: birth_date, intensity, duration, frequency, response_value, probability, confidence

### 4. API Routes (`app/api/v1/`)
Organizadas por domínio em subdiretórios:
```
app/api/v1/
├── auth/            # auth, admin, audit, consent
├── clinical/        # patients, consultations, professionals, episodes, scales, assessments, metrics, alerts, medications, reference
├── diagnostic/      # disorders, inferences
└── ml/              # training
```
Prefixo padrão: `/api/v1/...`

### 5. ML Pipelines (`app/ml/`)
Bayesian inference + ML training:
```
app/ml/
├── inference/               # Motor duplo (regras DSM-5-TR + Naive Bayes)
├── models/                  # 21 escalas psicométricas + SCALE_DISORDER_MAP
├── predictors/              # Personalidade (BFP/DT-12) + risco por escalas
├── evaluation/              # Avaliação de modelos
├── registry/                # MLflow registry wrapper
└── training/                # Pipeline de ML
    ├── trainer.py           # 4 objectives × 3 algorithms = 12 models
    ├── label_builder.py     # Label construction
    └── estimators.py        # Logistic Regression, Random Forest, XGBoost
```

## Key Guidelines

### Clinical Accuracy
- Always validate DSM-5-TR and ICD-11 code references
- Comment diagnostic criteria logic with official reference citations
- Ensure human-in-the-loop validation before inference results
- **Never** present probabilistic diagnoses as definitive diagnoses
- 19 Portuguese disorder names unified (ver AGENTS.md para lista completa)
- 3 camadas de validação clínica: Pydantic → IntegrityService → DB CHECK constraints

### Data Compliance (LGPD/GDPR)
- Use UUIDs everywhere instead of patient names/identifiers
- Separate PII storage from analytical data (patient_identity vs patient_profile)
- Fernet AES encryption for sensitive fields
- SHA-256 pseudonymization for CPF and email
- Consent management via `/api/v1/consent/`
- Audit logging with entity_id, operation_type, ip_address, user_agent

### Database Migrations
- Use Alembic for schema versioning (21 revisões lineares)
- Head: `e5f6a7b8c9d0` (professional assignments)
- Chain: initial → admin → trans_status → medications → reports → profession → notes → audit → alerts → CHECK constraints → authorities → assignments

### Seed Order (todos idempotentes)
1. `python db/seed.py` — Authorities + base data
2. `python scripts/seed_icd11.py` — CID-11 codes
3. `python scripts/seed_scales_groups.py` — Criteria groups + scales
4. `python scripts/seed_diagnostic_data.py` — DSM-5-TR criteria, exclusions, differentials

### Testing Strategy
- **pytest** fixtures for clinical datasets (mock patient profiles, symptoms)
- Mock external dependencies: UMLS lookups, DSM/ICD validation services
- Integration tests verify consultation → diagnosis workflow
- Unit tests for probability calculations and criteria evaluation

### Documentation Requirements
- Docstrings for all functions include clinical context
- README.md, DESENVOLVIMENTO.md, CLINICAL_MANUAL.md, STRUCTURE.md, QUICKSTART.md, SECURITY.md
- API documentation (FastAPI auto-docs: `/docs`, `/redoc`)
- All 19 disorder names in Portuguese (BR)

## Development Workflow

### When Creating New Files
1. **Models**: Add to `app/models/base.py` with UUID PKs and LGPD compliance notes
2. **Services**: Place business logic in `app/services/` with type hints
3. **Schemas**: Define Pydantic v2 models with `@field_validator` for clinical constraints
4. **Routes**: Use `app/api/v1/` structure with proper prefix `/api/v1/...`
5. **Tests**: Write pytest with fixtures; mock clinical data

### When Reviewing Code
- Check for DSM/ICD accuracy (19 Portuguese disorder names)
- Verify LGPD/GDPR compliance
- Ensure probabilistic results include confidence intervals
- Validate human-in-the-loop workflows
- Confirm 3-layer validation (Pydantic → Service → DB)

### When Writing Tests
- Use realistic clinical scenarios
- Mock external APIs (UMLS, clinical data services)
- Test edge cases: comorbidities, exclusionary criteria, differential diagnoses
- Validate both positive and negative diagnostic cases

## Common Tasks

### Task: Run Full Seed
1. `alembic upgrade head`
2. `python db/seed.py`
3. `python scripts/seed_icd11.py`
4. `python scripts/seed_scales_groups.py`
5. `python scripts/seed_diagnostic_data.py`

### Task: Train ML Pipeline
1. `python scripts/train_all.py` — treina 12 modelos (4 objectives × 3 algorithms)
2. Verificar no MLflow: `mlflow ui` → http://localhost:5000
3. Modelos promovidos para Production automaticamente

### Task: Check Clinical Data Integrity
1. `python scripts/check_integrity.py`
2. Relatório: `data/reports/clinical_integrity_report.json`
3. Verifica: pacientes, consultas, sintomas, escalas, inferências

### Task: Run DW ETL
1. `python -c "from app.etl.dw_loader import run_full_etl; run_full_etl()"`
2. Popula star schema: dim_disorder, dim_patient, dim_date, fact_*
