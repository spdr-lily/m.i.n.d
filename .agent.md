# M.I.N.D Project Agent Configuration

## Project Context
You are assisting with **M.I.N.D (Mental Intelligence & Network Data)** — a Clinical Decision Support System for mental health diagnosis. This is a hybrid Data Science + Psychiatry project focused on probabilistic diagnostic inference using DSM-5-TR and ICD-11 criteria.

**Current Phase:** MVP implementation with Python backend, PostgreSQL database, and FastAPI.

## Recommended Project Structure
```
project/
├── alembic/                    # Database migrations (Alembic versioning)
├── app/
│   ├── core/                   # config.py, constants.py, exceptions.py
```

## Code Patterns & Conventions

### 1. Service-Repository Pattern
- **Repositories** (`app/repositories/`): Pure data access, no business logic
  - Example: `patient_repository.py` with `get_by_uuid()`, `create_consultation()`
- **Services** (`app/services/`): Business logic, orchestration, inference
  - Example: `diagnosis_service.py` with `calculate_probabilistic_diagnosis()`

### 2. Pydantic Schemas (`app/schemas/`)
All API requests/responses use Pydantic for validation:
```python
from pydantic import BaseModel, Field
from uuid import UUID

class PatientCreateSchema(BaseModel):
    biological_sex: str
    gender_identity: str
    birth_date: date
    education_level: str
    
class DiagnosisResponseSchema(BaseModel):
    disorder_id: UUID
    confidence_score: float
    dsm_code: str
    icd_code: str
    is_differential: bool
```

### 3. SQLAlchemy Models (`app/models/`)
Use UUID as primary keys (LGPD compliance — no identifiers):
```python
from sqlalchemy import Column, String, Float, DateTime
from sqlalchemy.dialects.postgresql import UUID
import uuid

class Disorder(Base):
    __tablename__ = "disorders"
    id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    dsm_code = Column(String(20), unique=True)
    icd_code = Column(String(20), unique=True)
    name = Column(String(255))
```

### 4. API Routes (`app/api/`)
Organize by domain:
```
app/api/
├── __init__.py
├── patients.py         # POST /patients, GET /patients/{patient_id}
├── consultations.py    # POST /consultations, GET /episodes
├── diagnoses.py        # POST /calculate-diagnosis
└── assessment_scales.py # GET /scales/{scale_id}
```

### 5. ML Pipelines (`app/ml/`)
Bayesian inference and probabilistic calculations:
```
app/ml/
├── __init__.py
├── bayesian_network.py      # Probabilistic graphical models
├── inference_engine.py      # Conditional probability calculations
├── dsm_icd_mapper.py        # DSM-5-TR ↔ ICD-11 logic
└── criteria_evaluator.py    # Diagnostic criteria rule engine
```

## Key Guidelines

### Clinical Accuracy
- Always validate DSM-5-TR and ICD-11 code references
- Comment diagnostic criteria logic with official reference citations
- Ensure human-in-the-loop validation before inference results
- **Never** present probabilistic diagnoses as definitive diagnoses

### Data Compliance (LGPD/GDPR)
- Use UUIDs everywhere instead of patient names/identifiers
- Separate PII storage from analytical data
- Add encryption utilities for sensitive fields (consultation notes)
- Include LGPD retention/deletion comments in models
- Example comment: `# LGPD: PII separation - patient_id is UUID, no names stored`

### Database Migrations
- Use Alembic for schema versioning
- Track DSM-5-TR and ICD-11 code changes as migrations
- Include data validation migrations for disorder/criteria relationships

### Testing Strategy
- **pytest** fixtures for clinical datasets (mock patient profiles, symptoms)
- Mock external dependencies: UMLS lookups, DSM/ICD validation services
- Integration tests verify consultation → diagnosis workflow
- Unit tests for probability calculations and criteria evaluation

Example fixture:
```python
@pytest.fixture
def sample_patient():
    return {
        "biological_sex": "Female",
        "birth_date": "1990-05-15",
        "chief_complaint": "Persistent sadness"
    }
```

### Documentation Requirements
- Docstrings for all functions include clinical context
- README.md documents data sources (MIMIC-IV, UMLS, GBD Study)
- API documentation (FastAPI auto-docs: `/docs`, `/redoc`)
- ERD diagrams for database schema

## Development Workflow

### When Creating New Files
1. **Models**: Add to `app/models/` with UUID PKs and LGPD compliance notes
2. **Services**: Place business logic with type hints and clinical context
3. **Schemas**: Define Pydantic models with Field validation rules
4. **Routes**: Use FastAPI dependency injection for auth/repositories
5. **Tests**: Write pytest with fixtures; mock clinical data

### When Reviewing Code
- Check for DSM/ICD accuracy
- Verify LGPD/GDPR compliance comments
- Ensure probabilistic results include confidence intervals
- Validate human-in-the-loop workflows

### When Writing Tests
- Use realistic clinical scenarios
- Mock external APIs (UMLS, clinical data services)
- Test edge cases: comorbidities, exclusionary criteria, differential diagnoses
- Validate both positive and negative diagnostic cases

## External Data Sources
- **MIMIC-IV**: De-identified EHR data (MIT)
- **UCI ML Repository**: Clinical datasets
- **UMLS Metathesaurus**: ICD-11 ↔ DSM-5-TR mappings
- **GBD Study (IHME)**: Epidemiological priors for Bayesian calibration
- **NSDUH/CDC**: Baseline prevalence statistics

## Common Tasks

### Task: Add a New Disorder
1. Create migration in `alembic/versions/`
2. Define SQLAlchemy model in `app/models/disorders.py`
3. Add Pydantic schema in `app/schemas/diagnosis.py`
4. Implement repository methods in `app/repositories/disorder_repository.py`
5. Add service logic in `app/services/diagnosis_service.py`
6. Write pytest tests in `tests/integration/test_disorder_flow.py`
7. Create API route in `app/api/diagnoses.py`

### Task: Implement Diagnostic Criteria Evaluation
1. Define criteria in `app/ml/criteria_evaluator.py`
2. Map DSM/ICD rules with citations (e.g., "DSM-5-TR: F32.x")
3. Implement probability weighting in `app/ml/bayesian_network.py`
4. Test with `tests/unit/test_criteria_evaluation.py`
5. Document human validation requirements

### Task: Add Psychometric Assessment Scale
1. Add scale metadata to `app/models/assessment_scales.py`
2. Create Pydantic schema for responses in `app/schemas/assessments.py`
3. Implement scoring logic in `app/services/assessment_service.py`
4. Add API endpoint in `app/api/assessment_scales.py`
5. Test scoring accuracy with known benchmarks
